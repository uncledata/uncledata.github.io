Context, broken down

Context is five concrete things — and you already enforce all of them.

Structure

Where things live. How modules are organized.

Conventions

Naming, imports, error handling, typing.

Tooling

pytest, ruff, uv. The choices you've made.

Domain

What the app does. The key abstractions.

"Don't touch"

Legacy code, workarounds with a reason, the intentional weird.

09

A tale of two functions

Both work. Only one passes code review.

Correct, but contextless
def get_user(user_id): try: user = db.query(User).filter( User.id == user_id ).first() if user is None: raise Exception("not found") return { "id": user.id, "email": user.email, } except Exception as e: return {"error": str(e)}
Correct, and in your style
async def get_user( user_id: UUID, db: DBSession, ) -> UserOut: """Return a single user, or raise.""" user = await db.get(User, user_id) if user is None: raise UserNotFound(user_id) return UserOut.model_validate(user)

Same behavior. One file ships; the other gets six comments on the PR.

What it actually looks like

A real CLAUDE.md for a Python project.

CLAUDE.md · repo root
# Project: invoice-processor ## Stack Python 3.12, FastAPI, SQLAlchemy 2.0, Postgres Tests: pytest + pytest-asyncio Package manager: uv Linter: ruff ## Conventions - API endpoints return Pydantic models, never dicts - Use Annotated for dependency injection - Async everywhere — no sync DB calls - Errors raise from app/exceptions.py - Imports: stdlib → third-party → local
CLAUDE.md (continued)
## Testing - Fixtures in conftest.py, never inline - factory_boy for model factories - Integration tests hit a real test DB - Run: uv run pytest -x --tb=short ## Don't - Don't use datetime.now() — use app.utils.utc_now() - Don't add deps without checking pyproject.toml - Don't write raw SQL — use the ORM

Forty lines of plain markdown. That's the whole trick.

14

Same prompt, different context

Prompt: "add an endpoint to list invoices by status."

Without CLAUDE.md
@app.route("/invoices") def list_invoices(): status = request.args.get("status") rows = db.execute( f"SELECT * FROM invoices " f"WHERE status = '{status}'" ).fetchall() return jsonify([dict(r) for r in rows])

Wrong framework. Sync. Raw SQL. Dict response. SQL injection.

With CLAUDE.md
@router.get("/invoices", response_model=list[InvoiceOut]) async def list_invoices( status: InvoiceStatus, db: DBSession, ) -> list[InvoiceOut]: stmt = select(Invoice).where( Invoice.status == status ) rows = (await db.execute(stmt)).scalars().all() return [InvoiceOut.model_validate(r) for r in rows]

FastAPI. Async. ORM. Pydantic out. Typed. Your style.

15

Spec-driven prompting

Better prompts look like specs, not wishes.

Wish
add an endpoint for invoices

The AI guesses everything. You'll guess what it guessed.

Better
Add a GET /invoices endpoint that filters by status. Use our FastAPI + SQLAlchemy conventions in CLAUDE.md.

Names the route, the stack, points to the spec.

Spec-driven
## Goal List invoices by status. ## Contract - GET /invoices?status=... - Returns list[InvoiceOut] - 422 on bad status ## Tests first - happy path + 4xx - factory_boy fixtures

Goal · contract · tests. The AI now has a target.

16

The loop that works

Write the spec. Then the code.

  • 01
    Write a one-page spec
    Goal, contract, edge cases, tests. Markdown. Five minutes.
  • 02
    Ask the AI to critique it
    "What's missing? What would break?" — before any code is written.
  • 03
    Tests, then implementation
    Generate failing tests from the spec. Then ask for code that passes them.
  • 04
    Spec lives in the repo
    Commit it next to the code. Next session, the AI reads it again.
17

One step further

Skills are reusable instruction bundles — for the moves you do over and over.

"Add a new endpoint"

Router → schema → service → test. Always in that order.

"Write a migration"

Alembic autogen → review → write data backfill → smoke test.

"Triage a flaky test"

Re-run with seed → check for time/order coupling → pin or fix.

The CLAUDE.md is the 80/20. Skills are for after.

One file, two audiences

# How we write code here ## Imports Group: stdlib, third-party, local. Blank line between. Absolute imports only. from app.x import y, never from .x import y. ## Errors Raise from app/exceptions.py. Never bare except. HTTP errors are mapped at the boundary, not inline. ## Tests Every endpoint has a happy-path test and a 4xx test. Use factory_boy, not inline objects. No mock.patch on our own modules. ## When in doubt Ask in #backend-help. We don't bite.
For Copilot

Read at every completion. Now your tab-complete matches your code review.

For the new hire

First file they open on day one. Same content. Different reader.

For future-you

Six months later, when you've forgotten why imports were absolute.

Section 05 / 06

What still
breaks.

Honest about the limits. None of this turns AI into a senior dev.
23

The honest list

It will still confidently write wrong code. Plan for it.

Context windows aren't infinite

Large codebases still confuse it. CLAUDE.md is a primer, not the whole repo.

Domain nuance is hard to encode

"Why this customer is special" is not a markdown bullet.

Plausible-looking wrong code

It will hallucinate APIs that almost exist. Read the diff. Always.

Config files drift

Update conventions, forget the file, train on stale patterns. Treat it like code.

The promise isn't “AI writes your app.”
It's “AI writes the boring parts correctly instead of almost correctly.”
